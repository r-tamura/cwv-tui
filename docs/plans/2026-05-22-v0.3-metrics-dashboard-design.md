# cwv-tui v0.3 — Metrics Dashboard Design

**Status:** Validated (brainstormed 2026-05-22). Implementation plan to follow.

## Goal

Add CloudWatch Metrics support so cwv-tui works as an oncall / incident-triage TUI: open it and immediately see system state, drill from a metric spike to the relevant logs in a few keystrokes.

## Use case

The target user is on oncall. They want:
- A glance to confirm everything is healthy ("are any alarms red right now?").
- A short path from a metric spike to the relevant log events during an incident.
- A persistent dashboard they can leave open in a tmux pane.

## Decided constraints

| Topic | Decision |
|---|---|
| Configuration | Local YAML (`~/.config/cwv-tui/dashboards.yaml`) validated with Zod; `--config <path>` overrides |
| Landing screen | The default dashboard with an Alarms count banner above it |
| Chart → Logs | Each chart declares `logGroups: [...]` in YAML; `Enter` opens those groups in Insights, pre-filling the spike's time window |
| Layout | Vertical stack of charts inside the dashboard, scrollable with `j`/`k`/`Ctrl+D`/`Ctrl+U` |
| Time range | Shared across all charts on the dashboard. Default 1h. `t` opens the existing TimeRangePicker. |
| Auto-refresh | Every 30 s for metrics, 60 s for alarms. `p` pauses/resumes, `r` is a manual one-shot. |
| Charting library | `asciichart` (single series per chart in MVP) |
| Multi-dashboard | Supported; switch with `d`, default chosen by `defaultDashboard:` in YAML |
| Compatibility | If YAML is absent, cwv-tui boots into the v0.2 Logs-only experience and shows a helpful empty state on the Dashboard tab |

Out of scope for v0.3: multi-series overlay, CloudWatch dashboard JSON import, alarm history graphs, custom thresholds, color theming.

## Top-level UX

A new tab bar replaces the implicit "Log Groups is everything" model:

```
[ Dashboard ]  [ Alarms ]  [ Logs ]                  Active alarms: 2 ●
```

- `Tab` / `Shift+Tab` cycles the three top-level modes.
- Each mode owns its existing back-stack: `Esc` still pops within a mode; `Tab` switches mode.
- The Alarms count banner is rendered above the StatusBar in every mode.
  - 0 active: green.
  - ≥ 1 in `ALARM`: red, lists first few alarm names.
  - ≥ 1 in `INSUFFICIENT_DATA`: yellow.
- Pressing `Enter` on the banner jumps to the Alarms tab and highlights the first red alarm.

## YAML schema

```yaml
defaultDashboard: lambda-prod

dashboards:
  lambda-prod:
    title: Lambda Production
    charts:
      - title: Errors
        namespace: AWS/Lambda
        metric: Errors
        dimensions: { FunctionName: foo-api }
        stat: Sum
        height: 8                        # rows
        logGroups: [/aws/lambda/foo-api] # drill-down target
      - title: Duration p99
        namespace: AWS/Lambda
        metric: Duration
        dimensions: { FunctionName: foo-api }
        stat: p99
        height: 8
        logGroups: [/aws/lambda/foo-api]
```

Zod parses this at startup. Failures produce a friendly multi-line error and exit code 2. Missing optional fields take sensible defaults (`stat: Average`, `height: 8`, `logGroups: []`).

## AWS layer behavior

`getMetricSeries(client, spec, window)` issues a single `GetMetricDataCommand`. When `useMetricSeries` drives a whole dashboard, every chart's query is **batched into one request** (the API accepts up to 500 metric queries per call).

Period is derived from the window:

| Window | Period |
|---|---|
| 15 m | 60 s |
| 1 h | 60 s |
| 6 h | 5 m |
| 24 h | 15 m |
| 7 d | 1 h |

Returned `Series` is a list of `{ timestamp, value }` plus the resolved label. Missing data is kept as `null` so the chart shows gaps rather than zero.

`listAlarms` calls `DescribeAlarmsCommand`, with optional state filter; for the banner, only state counts matter.

## Hooks behavior

`useMetricSeries`:
- Inputs: `{ client, dashboard, window, refreshSec, paused }`.
- Internal `setInterval`; cleared on unmount and when `paused` flips true.
- Keeps the last successful per-chart series during in-flight refresh so charts don't blank out.
- Failures localize: one chart can show `[fetch failed]` while the others keep refreshing.

`useAlarmsSummary`:
- 60 s interval, summarizes by state for the banner.
- Detailed list is fetched on demand by AlarmsView (no second poller).

## API budget

Default usage with one user leaving the tool open: 30 s metrics + 60 s alarms = ~4,000 requests/hour. CloudWatch's `GetMetricData` free tier covers 1,000,000 metric data points per month; a 10-chart dashboard hitting that cadence stays well within free tier.

## Testing strategy

Pure logic gets the tightest tests; rendering tests are smoke-only.

| Layer | Coverage |
|---|---|
| `lib/config.ts` | YAML happy path, schema errors (missing field, wrong type), `--config` override resolution |
| `aws/metrics.ts` | `aws-sdk-client-mock` resolves a batched response; verify per-chart unwrap, period derivation, missing-data preservation |
| `aws/alarms.ts` | State filter, empty list |
| `hooks/useMetricSeries.ts` | `vi.useFakeTimers` to drive intervals, pause/resume, error-localization |
| `components/MetricChart.tsx` | Single snapshot test |
| `views/DashboardView.tsx` | Render with a one-chart dashboard, verify `Enter` dispatches the right Insights route |
| `views/AlarmsView.tsx` | Filter by state, banner-driven landing |

Total expected new tests: ~25. Existing v0.2 tests must continue to pass unchanged.

## Compatibility & rollout

- `cwv-tui` without a YAML file boots into v0.2 behavior. The Dashboard tab shows "No dashboards configured — see README" and a quickstart command (`cwv-tui --init-config` writing a starter YAML, optional in v0.3.1).
- The `Tab` key is new but only affects the new tab bar; existing keys (Esc, j/k, q, /, ?) are unchanged.

## Parallel implementation strategy

The track structure differs from v0.2 because the dependency graph is deeper. We use a short Track 0 to establish types and dependencies, then four tracks run in parallel against those types, and finally a small Track E integrates them into App / Help / README.

```
Track 0 (~30 min, sequential)
    │
    ├──> Track A   AWS layer + config
    ├──> Track B   MetricChart component
    ├──> Track C   Dashboard hook + view  (mocks Track A in tests)
    └──> Track D   Alarms hook + view     (mocks Track A in tests)
                       │
                       ▼
                 Track E (sequential)
                 navigation + app.tsx + HelpDialog + README + E2E smoke
```

### Track 0 — Foundation (single small PR, blocks others)

- Add deps: `asciichart`, `js-yaml`, `zod`, `@aws-sdk/client-cloudwatch`.
- Land all new type definitions in `src/types.ts` (`ChartSpec`, `Series`, `Alarm`, `DashboardConfig`, route additions).
- Create empty/skeleton files for the four implementation tracks so each can be imported by sibling tracks without TS errors.

### Track A — AWS layer + config (parallel)

- `src/aws/metrics.ts`, `src/aws/alarms.ts`, `src/lib/config.ts`.
- Full test coverage via `aws-sdk-client-mock` and YAML fixtures.

### Track B — Chart component (parallel)

- `src/components/MetricChart.tsx` using `asciichart`.
- One snapshot test.
- No runtime dependency on other tracks beyond `Series` type from Track 0.

### Track C — Dashboard view (parallel)

- `src/hooks/useMetricSeries.ts`, `src/views/DashboardView.tsx`.
- Tests `vi.mock` Track A's exports against the published interface.
- Reuses `useVimNav`, `TimeRangePicker`, `useTextInputLock` from v0.2.

### Track D — Alarms (parallel)

- `src/hooks/useAlarmsSummary.ts`, `src/views/AlarmsView.tsx`.
- Tests `vi.mock` Track A.

### Track E — Integration (sequential, after A–D)

- Extend `src/state/navigation.ts` with new routes.
- Wire `Tab`/`Shift+Tab`, banner placement, route rendering in `src/app.tsx`.
- Expand `src/components/HelpDialog.tsx` with Dashboard and Alarms sections.
- README: keymap rows, configuration example, screenshot section.
- E2E smoke: launch with a fixture YAML, assert initial dashboard renders, Enter on a chart pushes the Insights route with `logGroups` populated.

### Conflict surface

| File | Touched by |
|---|---|
| `package.json`, `pnpm-lock.yaml` | Track 0 only |
| `src/types.ts` | Track 0 only |
| `src/state/navigation.ts`, `src/app.tsx`, `src/components/HelpDialog.tsx`, `README.md` | Track E only |
| All other new files | Owned by exactly one of A/B/C/D |

So the four parallel tracks are conflict-free among themselves; all integration conflict goes to Track E by construction.

## Risks

1. **Track 0 design errors propagate to all four parallel tracks.** Mitigation: keep Track 0 to types and dependencies only, no logic. If a type is missing, a follow-up small PR is cheaper than a 4-way merge.
2. **`asciichart` output width.** The library wraps when the terminal is narrow. Mitigation: in `MetricChart` cap width to `process.stdout.columns - 4` and let it scale down.
3. **Refresh stampede when the user has many dashboards.** v0.3 only refreshes the active dashboard; switching dashboards triggers an immediate fetch.
4. **CloudWatch eventual consistency.** Metrics for the last minute often appear late. The default 1h window with 60 s period side-steps this; documented in README.

## Open questions deferred to v0.4

- Importing CloudWatch console dashboards via `GetDashboardCommand`.
- Saving / generating dashboards interactively from the TUI.
- Per-chart custom time ranges (the `[Risk]` cost of UI complexity wasn't worth MVP).
- Linking alarms to their dashboard chart so a red banner highlights which row.
